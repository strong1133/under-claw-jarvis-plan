# 00 · 루프 제어 (Loop Control) — 종료 조건·회차·안전장치

이 스킬은 베이스 `under-claw-jarvis-plan`을 **한 회차(iteration)** 로 돌리고, 그 위에
**독립 검수 게이트**를 씌운 외부 루프다. 이 파일은 루프의 제어 규칙을 정의한다.

> **베이스와의 관계**: 한 회차의 *내부 처리*는 베이스 스킬(이해→계획→구현→검수)을 **그대로** 따른다.
> 이 스킬은 그 위에 *외부 critic 루프*를 더한다. 베이스의 내부 리뷰(Phase 5)와 별개로,
> **구현·오케스트레이터와 분리된 검수 세션**이 회차 산출물을 채점한다(→ `30-reviewer`, `40-scoring`).

## 루프 엔지니어링 원칙 (best practice 요약)
1. **Generator–Critic 분리**: 만드는 세션과 평가하는 세션을 물리적으로 분리한다. 같은 세션의 자기평가는
   sycophancy(자화자찬) 편향이 있어 신뢰하지 않는다. → 검수자는 산출물과 원요구만 보고, 구현자의 자기변론은 보지 않는다.
2. **Fresh context per iteration**: 회차마다 구현 세션을 **새로 시작·종료**한다. 누적 컨텍스트의 anchoring/오염을 끊는다.
   회차 간 학습은 오케스트레이터가 **검수 피드백을 다음 회차 프롬프트로 전달**해 잇는다(Reflexion, 외부 메모리).
3. **결정적 종료 조건**: 종료는 주관이 아니라 **숫자**로 한다 — 검수 종합점수 **TARGET 이상**(기본 9.5).
4. **Bounded loop**: 무한 루프 금지. **max 회차 cap**과 **정체(plateau) 감지**로 반드시 멈출 수 있어야 한다.
5. **감사 가능성**: 회차마다 점수·gap·결정을 로깅한다(`<loop round N>` … `<loop verdict N: score=X.X>`).
6. **reward hacking 방지**: 검수 루브릭은 원요구에 고정한다. 검수자는 적대적·회의적으로, 불확실하면 낮게 준다.

## 종료 조건 (먼저 하나라도 충족하면 루프 종료)
| 조건 | 판정 | 결과 |
|---|---|---|
| **성공** | 종합점수 **≥ TARGET** | 루프 종료 → 최종 산출물 확정 + 마감 보고 |
| **회차 cap 도달** | 회차 = `MAX_ROUNDS`(기본 5) | 루프 중단 → 현재 최고점 산출물 + 미달 사유·잔여 gap을 사용자에 에스컬레이션 |
| **정체(plateau)** | 직전 2회차 대비 최고점 개선폭 **< 0.2** | 루프 중단 → 한계·가정·선택지를 사용자에 에스컬레이션 |
| **차단(BLOCK)** | 구현/검수가 근거 있는 `[BLOCK]`(불가능·모순 요구 등) | 루프 중단 → 쟁점을 좁혀 사용자에 질문 |

- `MAX_ROUNDS`는 입력으로 덮어쓸 수 있다(예: `/under-claw-jarvis-plan-loop {요구} --max-rounds 8`). 미지정 시 5.
- `--target`이 있으면 그 값을, 없으면 9.5를 `TARGET`으로 Intake에서 한 번 확정한다. TARGET은 0.0~10.0 범위의 숫자여야 하며 잘못된 값은 작업 전 `[BLOCK]`으로 반환한다. 사용자가 더 낮춰도 9.5가 기본 권장임을 1줄로 고지한다.

## 회차 절차 (오케스트레이터가 1회차마다 수행)
```
ROUND N:
  1. <loop round N> 로깅
  2. 구현 세션 생성(fresh): 베이스 under-claw-jarvis-plan 완주
       입력 = 원요구 + (N>1이면) 직전 검수의 gap 리스트·점수표
       → 구현자는 산출물 + 변경요약 + 자기점검을 오케스트레이터에 보고하고 세션 종료
  3. 검수 세션 생성(분리): 산출물 + 원요구 + 프로젝트 관례만 전달(구현자 자기변론 제외)
       → 40-scoring 루브릭으로 채점, 종합점수 + 차원별 점수 + 구체 gap 리스트 반환
  4. <loop verdict N: score=X.X> 로깅
  5. 종료 조건 평가:
       score >= TARGET → 성공 종료
       cap/plateau/BLOCK → 에스컬레이션 종료
       그 외 → gap 리스트를 다음 회차 입력으로 적재하고 ROUND N+1
```

## 회차 간 회귀(cross-round feedback)
검수가 드러낸 결함은 다음 회차 구현 입력으로 들어간다. 단 **같은 gap이 2회차 연속 미해결**이면
정체로 보고(설계·요구 자체의 문제일 수 있음), 베이스의 단계 회귀(이해/계획으로 back)를 다음 회차
구현 프롬프트에 **명시 지시**한다(예: "이 gap은 구현이 아니라 요구 해석 문제 → 이해 단계부터 재정렬").

## 상태 테이블 (매 회차 갱신·로깅)
| 회차 | 구현 산출 요약 | 종합점수 | 차원별(요구/정확/관례/품질) | 미해결 gap | 결정 |
|---|---|---|---|---|---|
| 1 | … | 8.1 | 3.2/2.4/1.6/0.9 | … | 계속 |
| 2 | … | 9.6 | 3.9/2.9/1.9/0.9 | 없음 | 종료(성공) |
