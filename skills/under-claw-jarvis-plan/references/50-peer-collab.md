# 50 · 멀티에이전트 협업 — 독립 병행 → 교차 검토 → 합의

> 멀티에이전트는 **스킬이 실행된 현재 세션을 기본**으로 한다.
> 별도 tmux/2-pane/외부 도구가 **없어도** 멀티에이전트로 동작한다.

## 기본 메커니즘 (어디서나 동작) — 현재 세션의 서브에이전트
사고가 필요한 단계(이해·계획·분석·설계 결정)는 **오케스트레이터(현재 세션의 Claude)가
서브에이전트를 띄워** 다음을 수행한다. 이것이 **기본값**이며 추가 환경이 필요 없다.
1. **독립 병행**: 같은 문제를 **서로 다른 관점의 서브에이전트 N개**(`Agent`, 또는 대규모면 `Workflow`)로
   동시에 분석/초안 작성하게 한다(에이전트끼리 서로 안 봄).
2. **교차 검토**: 오케스트레이터가 각 초안을 모아 **적대적으로 교차 검토**하거나, 검증 전용
   서브에이전트를 띄워 차이점·허점·놓친 케이스를 지적하게 한다.
3. **합의 합성**: 결과를 근거로 합성. 불일치가 안 좁혀지면 **양안 + 권장안**을 사용자에게 제시.

## 선택 강화 (환경에 있을 때만) — 제2 모델 peer
환경에 **다른 모델 peer**(예: Claude+Codex 2-pane처럼 별도 모델이 동등 작업자로 함께 있는 경우)가
있으면, 위 패턴을 **교차-모델**로 확장한다. 이질적 모델이라 사각지대가 달라 검증이 더 강해진다.
- peer가 **없으면 이 절은 건너뛴다** — 기본 메커니즘(서브에이전트)만으로 충분하다.
- peer와의 통신은 **환경이 제공하는 방식**을 쓴다(2-pane이면 pane 통신, 그 외엔 해당 수단).
  특정 도구에 의존하지 않는다.

## 동등(parity) 원칙 (서브에이전트·peer 공통)
- **기능 위계 금지**: "설계는 A, 구현은 B" 식 고정 분리 금지. 모두 이해·계획·분석·구현에 참여.
- **분담은 업무 영역 기준**: 기능이 아니라 **프로젝트/모듈/파일 경계**로 나눈다.
- **facilitator ≠ boss**: 오케스트레이터는 루프를 진행할 뿐 **결정 우위가 아니다.** 결론은 근거로.
- **다양성이 가치**: 관점(또는 모델)이 다를수록 더 잡는다. 독립 분석 후 교차 검증.

## 협업 태그 (peer와 메시지 주고받을 때)
`[REVIEW]` 검토요청 / `[ASK]` 질문 / `[DIFF]` 변경의도 / `[FYI]` 공유 / `[ACK]`·`[BLOCK]`
- 동일 파일 동시 수정 금지: 변경 직전 `[DIFF]` → 상대 `[ACK]` 후 진행.
- 동일 쟁점 3회 왕복 금지 → 차이점만 정리해 결론/선택지로.

## 비용 의식
독립 병행은 작업량이 늘어난다. **다양성이 값어치 있는 사고·분석·설계 결정**에 쓰고,
기계적 작업은 영역 분담으로 병렬화한다. 단순 질의는 굳이 여럿 돌리지 않는다.

## 구성 스킬 로깅 (협업에 참여한 모든 에이전트)
구성 스킬을 적용할 때마다 **호출 직전 한 줄 로깅**한다. 누가 했는지 표기:
`<{태그} 호출 · {주체}>` (주체 예: ORCHESTRATOR / SUBAGENT / 제2모델 peer 이름).
| 적용 모듈 | 로깅 태그 |
|-----------|-----------|
| 00-karpathy | `<karpathy 호출>` |
| 10-understand (Understand-Anything) | `<understand-anything 호출>` |
| 20-plan (Superpowers) | `<superpowers:plan 호출>` |
| 30-implement (Superpowers) | `<superpowers:implement 호출>` |
| 40-review (Superpowers) | `<superpowers:review 호출>` |
| 50-peer-collab | `<peer-collab 적용>` |
- **호출 로깅 생략 불가**(결과 로깅은 선택). 프로젝트 환경 스킬은 구성 스킬이 아니므로 로깅 대상 아님.
