# 50 · 멀티에이전트 협업 — 독립 병행 → 교차 검토 → 합의

> 멀티에이전트는 **스킬이 실행된 현재 세션을 기본**으로 한다.
> 별도 tmux/2-pane/외부 도구가 **없어도** 멀티에이전트로 동작한다.

## 기본 메커니즘 (어디서나 동작) — 현재 세션의 서브에이전트
사고가 필요한 단계(이해·계획·분석·설계 결정)는 **오케스트레이터(현재 세션의 Claude 또는 Codex)가
서브에이전트를 띄워** 다음을 수행한다. 서브에이전트 도구가 없으면 현재 세션 안에서 역할별 독립 패스를 분리해 같은 절차를 따른다.
1. **독립 병행**: 같은 문제를 **서로 다른 관점의 서브에이전트 N개**(Claude의 `Agent`/`Workflow`, Codex의 multi-agent/subagent 도구 등)로
   동시에 분석/초안 작성하게 한다(에이전트끼리 서로 안 봄). 도구가 없으면 관점 A/B/C를 분리해 순차 분석한다.
2. **교차 검토**: 오케스트레이터가 각 초안을 모아 **적대적으로 교차 검토**하거나, 검증 전용
   서브에이전트를 띄워 차이점·허점·놓친 케이스를 지적하게 한다.
3. **전원 합의 합성**: 결과를 근거로 합성(아래 고정 스키마). 참여 agent 중 하나라도 근거 있는 반대를 내면 진행하지 않는다. 불일치가 안 좁혀지면 **양안 + 권장안**을 제시.

## 합성 산출 스키마 (고정 — "양안+권장안"을 형식으로 못박는다)
교차 검토 후 합성은 **자유 서술이 아니라 아래 형식**으로 낸다. 합성 메커니즘을 손에 잡히게 한다.
```md
### 합의 (independent agreement)
- <두 관점이 일치한 결론들 — 그대로 채택>

### 불일치 (divergence table)
| 쟁점 | A 관점 | B 관점 | 근거 비교 | 판정 / 권장 |
|------|--------|--------|-----------|-------------|
| <…> | <A안> | <B안> | <무엇이 더 강한 근거인가> | <채택안 + 왜 / 못 좁히면 '양안 보존 + 권장: X'> |

### 미해결 가정 (open) — 다음 단계로 전달
- <확인 필요 항목 → 누가/어떻게 닫나>
```
- 판정은 **근거로** 내린다(다수결·연장자 아님). 못 좁히면 **양안을 둘 다 보존**하고 권장안을 표시.
- 이 표가 계획(20-plan) 설계 doc의 "채택 근거"·"미해결 가정" 필드로 그대로 흘러간다.

## 단계별 합의 게이트 (HARD-GATE — 추인 금지)
> 모든 사고 단계(이해·계획·구현·검수)는 아래 게이트를 통과해야 다음 단계로 넘어간다.
> **전원 유효 ACK 없이는 진행·완료 선언을 하지 않는다.** 한 모델의 단독 발의를 나머지가 추인만 하는 구조를 금지한다.

**1) 단계별 독립 산출 (independent draft)**
- 단계 시작 시 facilitator는 **스코프·질문만** 공유한다(자기 안을 먼저 노출하지 않는다 — 독립 오염 방지).
- 각 모델은 서로의 안을 **보지 않고** 독립 산출물을 제출한다. 사고 단계는 `drafts/<stage>_<agent>.md` 파일 제출을 권장하고, 경량 작업은 메시지로 갈음할 수 있다.
- 단계별 독립 산출 필수 항목:

  | 단계 | 각 모델이 독립 제출 |
  |---|---|
  | 이해 | 요구 요약 / 성공기준(Karpathy) / 영향범위·제약 / 미해결질문 (+brownfield 3자대조) |
  | 계획 | 접근법 / 인터페이스 시그니처 / 파일·모듈 경계+담당선언 / 검증방법 / 버린 대안 1개 |
  | 구현 | 담당 모듈 코드+단위테스트 / 비담당자는 계약 대비 독립 리뷰 |
  | 검수 | 계약충족 / 실행검증 결과 / 범위이탈 / 잔여리스크 (작업자 self-review는 불인정, 최종 검수자는 비작업자) |
- 제출을 마치면 `[DRAFT_READY]`를 전 pane에 전파한다.

**2) Read-Lock (독립·정독 보장)**
- **전원 `[DRAFT_READY]` 전에는 타인의 드래프트를 열람하지 않는다.** 도구로 물리 강제가 어려우므로 **행동 규율**로 준수한다.
- 이탈은 ⑷의 인용 검증(정독 증명)과, 가능한 환경에서는 `drafts/` 파일 mtime ↔ `[DRAFT_READY]` 전송시각의 **역전 검사**(미리 읽고 늦게 쓴 흔적 탐지)로 잡는다.

**3) 교차검토 + 합성**
- 전원 `[DRAFT_READY]` 후 상호 열람·적대적 교차검토. facilitator가 위 **합의 / 불일치표 / 미해결가정** 고정 스키마로 공식 합성을 발행한다.
- 합성자는 타 산출물을 **임의 대체 금지**. 누락 모델이 있으면 그 단계는 **미완료**다.

**4) 유효 ACK (빈 추인 차단)**
- 단독 `[ACK]`는 **무효**. 유효 ACK = `[ACK]` + 합성결정 N개 요약 + 동의근거 1개 + 잔여리스크 유무 + **(사고 단계) 상대 드래프트 핵심문구 1줄 인용(정독 증명)**.
- 요건 미충족 시 `[INVALID_ACK]`로 반려하고 재요청한다.

**5) 게이트 상태표 (한 칸이라도 비면 진행 금지)**

  | 모델 | 독립산출 | 교차검토 | 유효ACK |
  |---|:---:|:---:|:---:|
  | CLAUDE | | | |
  | CODEX | | | |
  | GEMINI | | | |
- 다음 단계 진입 조건 = **3개 독립산출물 실재 + 합성표 발행 + 3모델 유효 ACK + 진행을 막는 BLOCK/미해결 = 0**.
- 합의는 문구 일치가 아니라 **결정항목 일치**(목표 / 계약 / 변경범위 / 파일소유권 / 검증방법 / 리스크 처리)로 판정한다.

**6) 데드락 처리**
- 동일 쟁점 **3회 초과** 핑퐁 → 차이만 요약 → **Karpathy 최소·단순안 임시채택 + verify 실행결과로 정렬** → 그래도 미해결이면 **사용자에게 양안 + 권장**을 제시(에스컬레이션).

> 적용 범위: 1-pane 단독은 서브에이전트로 동일 게이트를 모사하고, 2-pane은 Claude+Codex 전원, 3-pane extra는 Claude+Codex+Gemini 전원에 적용한다.

## 선택 강화 (환경에 있을 때만) — 제2/제3 모델 peer
환경에 **다른 모델 peer**(예: Claude+Codex 2-pane, Claude+Codex+Gemini 3-pane처럼 별도 모델이 동등 작업자로 함께 있는 경우)가
있으면, 위 패턴을 **교차-모델**로 확장한다. 이질적 모델이라 사각지대가 달라 검증이 더 강해진다.
- peer가 **없으면 이 절은 건너뛴다** — 기본 메커니즘(서브에이전트)만으로 충분하다.
- peer와의 통신은 **환경이 제공하는 방식**을 쓴다(2/3-pane이면 pane 통신, 그 외엔 해당 수단).
  특정 도구에 의존하지 않는다.

## 동등(parity) 원칙 (서브에이전트·peer 공통)
- **기능 위계 금지**: "설계는 A, 구현은 B" 식 고정 분리 금지. 모두 이해·계획·분석·구현에 참여.
- **분담은 업무 영역 기준**: 기능이 아니라 **프로젝트/모듈/파일 경계**로 나눈다.
- **facilitator ≠ boss**: 오케스트레이터는 루프를 진행할 뿐 **결정 우위가 아니다.** 결론은 근거로.
- **전원 동의 원칙**: 이해, 계획, 구현 방향, 파일 변경 경계, 검수 통과, 완료 선언은 현재 참여 중인 모든 agent의 명시적 동의가 필요하다. 2-pane은 Claude+Codex 전원, 3-pane extra는 Claude+Codex+Gemini 전원이다.
- **작업자/검수자 분리**: 목표와 결과물 검증은 작업 agent와 다른 agent가 맡는다. 3-pane에서는 가능하면 작업자를 제외한 두 agent가 교차 검수한다.
- **다양성이 가치**: 관점(또는 모델)이 다를수록 더 잡는다. 독립 분석 후 교차 검증.

## 협업 태그 (peer와 메시지 주고받을 때)
`[REVIEW]` 검토요청 / `[ASK]` 질문 / `[DIFF]` 변경의도 / `[FYI]` 공유 / `[ACK]`·`[BLOCK]`
`[DRAFT_READY]` 독립 산출물 제출 완료(이후 상호 열람 가능) / `[INVALID_ACK]` 빈 추인 반려
- 동일 파일 동시 수정 금지: 변경 직전 `[DIFF]` → 참여 agent 전원 `[ACK]` 후 진행.
- 한 agent라도 `[BLOCK]`하면 진행하지 않고 `[DIFF]`/`[ASK]`로 쟁점을 좁힌 뒤 다시 합의한다.
- 동일 쟁점 3회 왕복 금지 → 차이점만 정리해 결론/선택지로.
- **단독 `[ACK]` 무효** — 유효 ACK는 위 "단계별 합의 게이트 ⑷" 형식(요약+근거+리스크+(사고단계)인용)을 갖춰야 게이트에 계상된다.

## 비용 의식
독립 병행은 작업량이 늘어난다. **다양성이 값어치 있는 사고·분석·설계 결정**에 쓰고,
기계적 작업은 영역 분담으로 병렬화한다. 단순 질의는 굳이 여럿 돌리지 않는다.

## 구성 스킬 로깅 (협업에 참여한 모든 에이전트)
구성 스킬을 적용할 때마다 **호출 직전 한 줄 로깅**한다. 누가 했는지 표기:
`<{태그} 호출 · {주체}>` (주체 예: ORCHESTRATOR / SUBAGENT / 제2모델 peer 이름).
| 적용 모듈 | 로깅 태그 |
|-----------|-----------|
| 00-karpathy | `<karpathy 호출>` |
| 10-understand (Understand-Anything) | `<understand-anything 호출>` |
| 20-plan (Superpowers) | `<superpowers:plan 호출>` |
| 30-implement (Superpowers) | `<superpowers:implement 호출>` |
| 40-review (Superpowers) | `<superpowers:review 호출>` |
| 50-peer-collab | `<peer-collab 적용>` |
- **호출 로깅 생략 불가**(결과 로깅은 선택). 프로젝트 환경 스킬은 구성 스킬이 아니므로 로깅 대상 아님.
