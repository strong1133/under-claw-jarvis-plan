# 20 · Phase 3 — 계획 (Plan)

> 출처: Superpowers brainstorming + writing-plans 발췌·적응 (obra/superpowers, MIT).

## 3-1. 브레인스토밍 (설계 정렬)
구현 액션을 취하기 전에 **설계를 제시하고 합의**한다(소프트 게이트 — under-claw-jarvis-plan은
대부분 자율 결정하되, 되돌리기 어려운/대형 변경은 설계 합의 후 진행).

1. 이해 단계 산출(요구·코드지도·가정)을 바탕으로 **접근법 2~3개**를 서로 다른 관점
   (MVP우선 / 리스크우선 / 사용자우선)으로 도출.
2. 각 접근법의 트레이드오프를 명시하고 **권장안**을 고른다(이유 포함).
3. council 내부에서 judge로 평가·점수화 → 승자 채택, 차선의 좋은 아이디어 접목.
4. 적대적 skeptic으로 설계 허점·실패모드를 사전 공격(Karpathy 1).

> **독립 병행**: 서로 다른 관점의 서브에이전트가 각자 접근법/설계 초안을 내고 교차 검토 후 합성한다
> (→ `references/50-peer-collab.md`). 설계는 한 에이전트의 전유물이 아니다. 불일치는 양안+권장안으로. 제2모델 peer가 있으면 교차-모델로.

## 3-2. 설계 문서화 (파일기반 메모리 · 고정 스키마)
합의된 설계를 **설계 doc으로 저장**한다 — 세션을 넘는 영속화 수단(경량). **불릿 나열이 아니라 아래 고정 스키마**로 쓴다.
- 경로: 작업 프로젝트 기준 `docs/under-claw-jarvis-plan/specs/<날짜>-<주제>-design.md`
  (프로젝트에 docs/가 없으면 적절한 기록 폴더에 둔다).

```md
# <주제> 설계 doc  (<날짜>)
1. 요구 & 성공기준        # 검증 가능한 형태(Karpathy 4). 각 기준에 [verify: 방법]
2. 채택 접근법 & 근거      # 비교한 대안 / 채택 이유 / 버린 이유(트레이드오프)
3. 변경 범위 & 파일        # 건드릴 파일·경계 / 건드리지 않을 영역(제약)
4. 프로젝트 간 계약 영향    # API 경로·메서드·DTO·MCP·인터페이스 변화(다중 프로젝트면 필수)
5. 리스크 & 미해결 가정     # 각 리스크에 완화책 / 가정은 '확인 필요'로 표시
6. 검증 방법              # 빌드/테스트/수동확인 — 성공기준과 1:1 대응
7. task 분할             # 3-3 의 번호 task 리스트(여기로 이어짐)
```
- 각 필드는 **비우지 않는다**(해당 없음이면 "없음"). placeholder/모순/모호/범위이탈 self-review(Karpathy 2·3).
- brownfield면 3자 대조표(10-understand)를 1·3에 반영.

## 3-3. 작업 계획 (실행 분할)
설계를 **독립적이고 작은 task로 분해**한다.
- 각 task: 정확한 변경 명세 + **검증 방법**(Karpathy 4) + 건드릴 파일 경계.
- task끼리 **파일 충돌이 없게** 경계를 나눈다(병렬/영역 배분 대비).
- 의존 순서를 표시(무엇이 무엇을 막는지).
- 산출: 번호 매긴 task 리스트 → 구현 단계(30)로 전달.
