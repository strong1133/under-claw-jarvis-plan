# 30 · Phase 4 — 구현 (Implement)

> 출처: Superpowers subagent-driven-development 발췌·적응 (obra/superpowers, MIT).
> 핵심 공식: **task별 fresh 작업자 + 2단계 리뷰(스펙 → 품질) = 빠르고 높은 품질**.

## 작업자 배분

### 기본 — 현재 세션 서브에이전트
- task마다 **fresh 서브에이전트**(`Agent`)를 띄워 구현. 컨텍스트는 task별로 격리.
- 병렬 변경이 충돌하면 worktree 격리(`isolation: 'worktree'`) 또는 `Workflow`로 파이프라인.
- **교차 리뷰**: 한 산출물은 **다른(검증 전용) 서브에이전트**가 스펙→품질 2단계 리뷰.
- **전원 동의**: 내부 council/subagent의 이해, 구현 방향, 완료 판정 합의가 있어야 다음 단계로 간다.

### 선택 — 제2 모델 peer가 있을 때 (→ `references/50-peer-collab.md`)
- 환경에 다른 모델 peer가 있으면 **영역(프로젝트/모듈/파일) 분담**으로 병렬 구현하고
  **작업하지 않은 다른 peer가 교차 리뷰**(이질적 모델 = 더 강한 검증). 기능 위계(설계=A/구현=B) 금지.
- 2-pane은 Claude+Codex 전원 동의, 3-pane extra는 Claude+Codex+Gemini 전원 동의가 필요하다.
- peer 통신은 환경이 제공하는 방식. **동일 파일 동시 수정 금지** — 변경 직전 `[DIFF]` → 참여 agent 전원 `[ACK]`.

## task별 워크플로
1. 작업자에게 **task 전문 + 맥락**을 전달(스펙 완비 시 더 빠른 모델 사용 가능).
2. 작업자가 구현 → 자체 테스트 → self-review.
3. **스펙 준수 리뷰**(작업자가 아닌 리뷰어): 요구를 다 만족하는가 — 빠진 것/넘친 것 없는가(Karpathy 2·3).
   - 통과 못 하면 작업자가 고치고 재리뷰.
4. **품질 리뷰**(작업자가 아닌 리뷰어): 설계·패턴·유지보수성. **스펙 통과 전엔 품질 리뷰 시작 금지.**
   - 문제 있으면 고치고 재리뷰. "이 정도면 됨" 금지.
5. 참여 agent 전원 `[ACK]` 후 task 완료 표시.

## 작업자 상태 신호 처리
- `DONE` → 스펙 리뷰로.
- `DONE_WITH_CONCERNS` → 우려 읽고 정확성/범위 문제는 리뷰 전 해결.
- `NEEDS_CONTEXT` → 빠진 정보 보강 후 재투입.
- `BLOCKED` → 무언가 바꾼다(컨텍스트/모델/범위) 또는 에스컬레이션.

## 절대 금지(레드 플래그)
- 리뷰 건너뛰기 / 미해결 문제로 진행 / 보호 브랜치에 합의 없이 시작 /
  구현 작업자 여러 명 동시 투입(단독 Claude 시) / 스펙 통과 전 품질 리뷰.

### [3-Pane 진짜 독립 합의 게이트 DoD]
- **독립 산출물 (구현자)**: 계약된 파일 경계 규칙에 맞게 본인 담당 모듈의 완결된 코드 파일 및 단위 테스트 코드를 개별 작성하여 제출한다. (동일 파일 동시 수정 절대 금지. 변경 전 `[DIFF]` -> 전원 `[ACK]`)
- **독립 산출물 (비구현 검수자)**: 구현된 타인의 모듈 코드에 대해 계약 사양 및 Simplicity First 관점을 적대적으로 정독한 뒤, 계약 대비 리뷰 소견을 독자 제출한다.
