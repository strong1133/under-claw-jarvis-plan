# 10 · Phase 2 — 이해 (Understand)

목표: **요구를 정확히 정렬**하고 **코드 구조를 충분히 파악**한 뒤에야 계획으로 넘어간다.
Karpathy 1번(가정 금지)을 강하게 적용한다.

## 두 종류의 이해를 구분해서 라우팅한다

### A. 요구 이해 (소크라테스식 — 사람과 정렬)
선택 프로젝트의 `CLAUDE.md`/`AGENTS.md`/`docs/`를 먼저 읽어 **스스로 최대한 파악**한 뒤,
남는 모호함만 질문한다. 무작정 묻지 않는다.
- 핵심 목표·성공 기준 식별 → 검증 가능한 형태로(Karpathy 4).
- 해석이 갈리면 선택지로 제시(Karpathy 1).
- 이해한 바를 1~3줄로 요약해 확인.

> **이해·분석은 독립 병행으로**: 서로 다른 관점의 서브에이전트로 각자 분석 → 교차 검토 → 합의
> (→ `references/50-peer-collab.md`). 제2모델 peer가 있으면 교차-모델로 확장. 한 에이전트가 독점하지 않는다.

### B. 코드 구조 이해 (기계적 — 멀티에이전트 스캔)
프로젝트가 **낯설거나 크거나, 변경 영향 범위가 불확실**하면 구조 매핑을 수행한다.
- **Understand-Anything 플러그인이 설치돼 있으면** 그 `/understand`(증분 분석 → knowledge-graph)
  와 `/understand-diff`(변경 영향 분석)를 우선 활용한다.
- 미설치거나 가벼운 파악이면 **council 직접 fan-out**: reader 에이전트들을 병렬로 띄워
  레이어(API/Service/Data/UI/Util), 진입점, 핵심 도메인, 의존 경계를 병렬 파악하고
  결론만 모은다(파일 덤프 금지).

#### 서브에이전트 출력 규율 (덤프 금지 — 프롬프트에 못박는다)
fan-out 에이전트를 띄울 때 **출력 형식을 프롬프트에서 강제**한다:
- "**질문 번호별 결론을 각 ≤3줄**로. 근거는 `파일:라인` 1~2개만. 전체 파일/긴 코드 붙여넣기 금지."
- "사실만(추측 금지), 없으면 '없음'. 표/목록은 핵심 식별자만."
이렇게 해야 오케스트레이터가 결론을 받지, 파일 덤프에 파묻히지 않는다(스킬 원칙 전파).

> 판단 기준: "이 변경이 어디까지 번지는지 자신 있게 말할 수 있는가?"
> 아니오 → 구조 매핑(B). 예 → A만으로 충분.

## 다중 프로젝트 처리
선택된 프로젝트가 여러 개면(예: api + api-core + web), **프로젝트 간 계약(인터페이스/DTO/API
경로)** 을 먼저 식별한다. 한 프로젝트의 변경이 다른 프로젝트에 주는 파급을 이해 단계에서 명시.

## 수정·리팩토링(brownfield) 모드 — 최초요구 ↔ 현재구현 ↔ 교정요청 3자 대조
Intake에서 brownfield로 판정되면(기존 구현 + 교정 요청 존재), 새로 짜기 전에 **현 상태를 진단**한다.
1. 각 대상 프로젝트의 **현재 구현 상태**를 멀티에이전트로 병렬 파악(관련 엔티티/흐름/엔드포인트/연동).
2. **3자 대조표**를 만든다(범례: ✅이미 부합 / ⚠️이탈·과구현 / ❓확인 필요 / ➕누락):
   - 최초 요구사항이 의도한 것
   - 현재 구현이 실제로 하는 것
   - 교정 요청이 바꾸라는 것
   → 여기서 **과구현 / 요구 이탈 / 누락 / 잘못된 결합**을 항목별로 식별.
   - **"이미 부합" 패스트패스**: 현재 구현이 교정요청을 이미 충족하는 항목은 ✅로 빠르게 닫고
     변경 후보에서 제외한다. 시간·집중은 ⚠️/❓/➕에 쏟는다. brownfield라고 다 갈아엎지 않는다.
   - 대조 후 **한 줄 총평**(예: "현재 구현이 N% 이미 충족 → 외과적 정리 범위")으로 수정 규모를 못박는다.
3. **제약 경계 고정**: "변경 금지"로 지정된 영역(예: 이미 서비스 중인 구조)은 건드리지 않는다.
4. 방침: **기존 구현 유지에 집착하지 않는다.** 요구를 이탈한 코드는 적극 변경·제거한다.
   (Karpathy 3의 "외과적 변경"은 *무관한* 코드 보호용 — 요구 이탈 코드는 교정 대상이다.)
5. 도메인 분리/SRP/과한 추상화 금지 등 **설계 제약**을 이해 단계에서 명문화해 계획으로 전달.

## 산출
- 정렬된 요구 + 성공 기준(검증 가능)
- 관련 코드 지도(영향 범위, 변경 후보 파일, 프로젝트 간 계약)
- 미해결 가정/리스크 목록 → 계획 단계로 전달

### [3-Pane 진짜 독립 합의 게이트 DoD]
- **독립 산출물**: 각 에이전트는 서로의 안을 참고하지 않고 `drafts/understand_<에이전트>.md` 파일(또는 독립 메시지)을 독자 작성해 제출하고 `[DRAFT_READY]`를 전파한다. 전원 완료 전까지는 타 드래프트 열람 금지(Read Lock).
- **필수 포함 필드**:
  1. 요구사항 요약 (1줄)
  2. 성공 기준 (Karpathy 4에 기반한 검증 가능한 형태)
  3. 관련 코드 지도 및 변경 범위 영향 파악
  4. 미해결 질문 및 제약 조건 목록 (+Brownfield일 경우 3자 대조표 필수)
- **교차 합성**: 전원 완료 후 촉진자가 3개 안을 대조하여 합의/불일치표(Divergence Table)/미해결가정 문서를 발행하고, 3모델이 모두 유효 ACK를 승인해야 다음 단계로의 게이트가 오픈된다.
