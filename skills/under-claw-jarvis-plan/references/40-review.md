# 40 · Phase 5 — 검수 (Review)

> 출처: Superpowers requesting-code-review + verification-before-completion 발췌·적응 (MIT).

## 5-1. 전체 코드 리뷰
모든 task 완료 후, **전체 구현을 종합 리뷰**한다(개별 task 리뷰와 별개).
- 설계 doc(20-plan) 대비 일치 여부 — 빠진 요구/범위 이탈 점검.
- 프로젝트 간 계약(API 경로/메서드/DTO) 일치 — 다중 프로젝트 변경 시 필수.
- Karpathy 2·3 위반(과한 추상화, 무관한 변경, orphan) 적대적 점검.
- 종합 리뷰도 **검증 전용 서브에이전트**로 교차 검증한다. 제2모델 peer가 있으면 작업하지 않은 모델에게 독립 리뷰 요청(이질적 관점 = 더 강한 검증).
  2-pane은 상대 agent가, 3-pane extra는 작업자를 제외한 두 agent가 가능하면 모두 리뷰한다.

## 5-2. 검증 (말이 아니라 실행으로)
"됐다"고 말하기 전에 **실제로 동작을 확인**한다(Karpathy 4).
- 빌드/타입체크/린트 실행.
- 관련 테스트 실행(없으면 핵심 경로 재현 테스트 작성 후 통과).
- 성공 기준(이해 단계에서 정의)과 대조 — 하나씩 충족 확인.
- 실행 검증 결과는 작업 agent가 아닌 다른 agent가 확인한다.
- 실패하면 숨기지 말고 출력과 함께 보고하고 고친다.

## 5-3. 마감
- 변경 요약(무엇을·왜)을 결론 중심으로 사용자에게 전달(핑퐁 중계 금지).
- 설계 doc 갱신(실제 구현과 차이가 있으면 반영) — 파일기반 메모리 최신화.
- 참여 agent 전원 동의가 확인된 뒤에만 완료로 선언한다.
- 커밋/푸시는 사용자 요청 시에만. 푸시 전 저장소 정책상 민감정보 검사 규칙이 있으면 준수.
- 남은 리스크/후속 작업을 명시.

## 5-4. 완료 정리 문서 (Intake에서 경로를 받은 경우)
사용자가 정리 문서 산출 경로(예: `~/Desktop/<주제>-spec.md`)를 지정했으면 **완료 후 그 경로에 작성**한다.
- 포함: 최종 요구사항, 채택 설계와 근거, **프로젝트별 변경 요약**, 데이터 모델/엔티티,
  요청 흐름(동기/비동기 경계 명시), 프로젝트 간 계약(API/MCP/DTO), 제약 준수 내역, 검증 결과,
  남은 리스크/후속.
- brownfield면 **변경 전→후 대비**(무엇을 왜 바꿨는지, 제거한 과구현)를 명시.
- 설계 doc(20-plan)과 중복이면, 정리 문서는 사용자용 최종본으로 깔끔히 통합한다.

## 마감 체크리스트
- [ ] 성공 기준을 실행으로 검증했는가 (작업자 자체 self-review는 승인으로 불인정하며, 실행 검증 결과 확인자는 반드시 작업자가 아닌 타인 모델이 수행함)
- [ ] 검증·검수자가 작업자와 분리되었는가
- [ ] 참여 agent 전원이 완료 판정에 동의했는가
- [ ] 설계 doc과 구현이 일치하는가
- [ ] 프로젝트 간 계약이 깨지지 않았는가
- [ ] Karpathy 4원칙(가정/단순/외과적/검증)을 지켰는가
- [ ] 사용자에게 결론과 근거만 간결히 전달했는가

### [3-Pane 진짜 독립 합의 게이트 DoD]
- **독립 산출물**: 각 에이전트는 검수 단계 완료 전 `drafts/review_<에이전트>.md` 파일(또는 독립 메시지)을 독자 작성해 제출하고 `[DRAFT_READY]`를 전파한다. 전원 완료 전까지는 타 검수 드래프트 열람 금지(Read Lock).
- **필수 포함 필드**:
  1. 계약 사양(CONTRACT.md 등)에 완벽히 충족하는지 여부
  2. 실제 빌드, 테스트(Ran N tests, OK), 수동 실행 검증 결과 기록 교차 확인
  3. 요구사항 범위를 넘어선 오버엔지니어링/범위 이탈 변경이 없는지 여부
  4. 남은 잔여 리스크 분석 요약
- **최종 유효 ACK 게이트**: 3모델 전원이 위 필수 필드가 충족된 검수 드래프트를 올리고, 교차 대조 결과에 찬성하는 최종 유효 ACK를 승인할 때만 전원동의 마감 처리(PASS)를 수행한다.
