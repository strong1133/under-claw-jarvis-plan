# 40 · Phase 5 — 검수 (Review)

> 출처: Superpowers requesting-code-review + verification-before-completion 발췌·적응 (MIT).

## 5-1. 전체 코드 리뷰
모든 task 완료 후, **전체 구현을 종합 리뷰**한다(개별 task 리뷰와 별개).
- 설계 doc(20-plan) 대비 일치 여부 — 빠진 요구/범위 이탈 점검.
- 프로젝트 간 계약(API 경로/메서드/DTO) 일치 — 다중 프로젝트 변경 시 필수.
- Karpathy 2·3 위반(과한 추상화, 무관한 변경, orphan) 적대적 점검.
- 종합 리뷰도 **검증 전용 서브에이전트**로 교차 검증한다. 제2모델 peer가 있으면 그 모델에게 독립 리뷰 요청(이질적 관점 = 더 강한 검증).

## 5-2. 검증 (말이 아니라 실행으로)
"됐다"고 말하기 전에 **실제로 동작을 확인**한다(Karpathy 4).
- 빌드/타입체크/린트 실행.
- 관련 테스트 실행(없으면 핵심 경로 재현 테스트 작성 후 통과).
- 성공 기준(이해 단계에서 정의)과 대조 — 하나씩 충족 확인.
- 실패하면 숨기지 말고 출력과 함께 보고하고 고친다.

## 5-3. 마감
- 변경 요약(무엇을·왜)을 결론 중심으로 사용자에게 전달(핑퐁 중계 금지).
- 설계 doc 갱신(실제 구현과 차이가 있으면 반영) — 파일기반 메모리 최신화.
- 커밋/푸시는 사용자 요청 시에만. 푸시 전 저장소 정책상 민감정보 검사 규칙이 있으면 준수.
- 남은 리스크/후속 작업을 명시.

## 5-4. 완료 정리 문서 (Intake에서 경로를 받은 경우)
사용자가 정리 문서 산출 경로(예: `~/Desktop/<주제>-spec.md`)를 지정했으면 **완료 후 그 경로에 작성**한다.
- 포함: 최종 요구사항, 채택 설계와 근거, **프로젝트별 변경 요약**, 데이터 모델/엔티티,
  요청 흐름(동기/비동기 경계 명시), 프로젝트 간 계약(API/MCP/DTO), 제약 준수 내역, 검증 결과,
  남은 리스크/후속.
- brownfield면 **변경 전→후 대비**(무엇을 왜 바꿨는지, 제거한 과구현)를 명시.
- 설계 doc(20-plan)과 중복이면, 정리 문서는 사용자용 최종본으로 깔끔히 통합한다.

## 마감 체크리스트

- [ ] 설계 doc과 구현이 일치하는가
- [ ] 프로젝트 간 계약이 깨지지 않았는가
- [ ] Karpathy 4원칙(가정/단순/외과적/검증)을 지켰는가
